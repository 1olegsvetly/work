РОАДМАП И ЧЕК-ЛИСТ ПО БЕЗОПАСНОСТИ САЙТА KONODEN
ВВЕДЕНИЕ
Безопасность веб-сайта в 2025–2026 годах — это не просто опция, а обязательное условие для любого бизнеса. Кибератаки растут, законодательство ужесточается, а утечки данных могут стоить миллионы и репутацию компании. Даже небольшой лендинг или сайт-визитка уже находится в зоне риска.
Сайт Konoden работает с персональными данными клиентов (имена, Telegram, email, ссылки на каналы) и финансовой информацией (заявки на подключение монетизации). Утечка этих данных может привести не только к финансовым потерям, но и к потере доверия клиентов — а значит, и бизнеса.
Данный документ содержит:
Анализ актуальных угроз для веб-приложений в 2025–2026 годах
Роадмап по внедрению безопасности — 6 этапов с конкретными задачами и сроками
Детальный чек-лист из 100+ пунктов для проверки и аудита безопасности
ЧАСТЬ 1: АНАЛИЗ АКТУАЛЬНЫХ УГРОЗ (OWASP TOP 10 2025)
OWASP (Open Web Application Security Project) выпустил обновлённый рейтинг Top 10 самых критичных веб-уязвимостей на 2025 год. Вот актуальный список угроз, которые необходимо учитывать при разработке и защите сайта Konoden:
1. A01:2025 — Broken Access Control (Нарушение контроля доступа)
Статус: 🥇 #1 самая критичная уязвимость
Нарушение контроля доступа происходит, когда пользователи получают доступ к объектам или функциям, к которым у них нет разрешения. В среднем 3.73% протестированных приложений имеют хотя бы одну уязвимость этого типа.
Пример для Konoden: Если злоумышленник сможет получить доступ к папке /data/ с JSON-файлами (где хранятся заявки клиентов) через прямое обращение к URL — это классический Broken Access Control.
Что делать:
Защитить папку /data/ через .htaccess (Deny from all)
Все запросы к данным должны проходить через PHP-обработчики с проверкой сессии
Использовать принцип наименьших привилегий — каждый пользователь имеет доступ только к тому, что ему действительно нужно
2. A02:2025 — Security Misconfiguration (Ошибки конфигурации безопасности)
Статус: ⬆️ Поднялся с #5 в 2021 до #2 в 2025
Ошибки конфигурации — одна из самых распространённых уязвимостей. 3.00% протестированных приложений имеют такие проблемы.
Примеры для Konoden:
Неправильные права доступа к файлам и папкам
Включённый display_errors на продакшене (раскрывает путь к файлам)
Стандартные учётные данные (admin/admin) в админ-панели
Отсутствие HTTPS или неправильная настройка SSL
Что делать:
Отключить display_errors на продакшене
Сменить пароль администратора (не оставлять admin/admin)
Настроить HTTPS с правильными заголовками безопасности
Регулярно проверять конфигурацию сервера
3. A03:2025 — Software Supply Chain Failures (Уязвимости в цепочке поставок)
Статус: 🆕 Новая категория в 2025
Это расширение предыдущей категории «Уязвимые и устаревшие компоненты». Включает уязвимости в зависимостях, библиотеках, плагинах и инфраструктуре сборки.
Пример для Konoden:
Использование устаревшей версии PHP (например, 7.4, в которой есть известные CVE)
Уязвимости в библиотеках Chart.js, TinyMCE (WYSIWYG-редактор)
Устаревшие зависимости в админ-панели
Что делать:
Регулярно обновлять PHP до актуальной стабильной версии
Мониторить уязвимости в используемых библиотеках
Использовать только проверенные и поддерживаемые компоненты
4. A04:2025 — Cryptographic Failures (Криптографические уязвимости)
Статус: ⬇️ Спустился с #2 в 2021 до #4 в 2025
Среднестатистическое приложение имеет 3.80% вероятность наличия криптографической уязвимости.
Пример для Konoden:
Хранение паролей в открытом виде (в админ-панели)
Отсутствие HTTPS (передача данных по незащищённому каналу)
Слабые алгоритмы хеширования
Что делать:
Использовать password_hash() с алгоритмом PASSWORD_DEFAULT для хранения паролей
Обязательно HTTPS с актуальным SSL-сертификатом
Не хранить чувствительные данные в открытом виде
5. A05:2025 — Injection (Инъекции)
Статус: ⬇️ Спустился с #3 в 2021 до #5 в 2025
Включает широкий спектр атак: от XSS (высокая частота/низкое влияние) до SQL-инъекций (низкая частота/высокое влияние).
Примеры для Konoden:
XSS (Cross-Site Scripting): Злоумышленник вставляет JavaScript-код в поле формы, который выполняется в браузере других пользователей
XSS в JSON: Если данные из JSON выводятся без экранирования — это путь к XSS-атаке
PHP Object Injection: Небезопасное использование unserialize() может привести к выполнению произвольного кода
Что делать:
Использовать htmlspecialchars() для экранирования вывода в HTML
Для сложных случаев использовать HTML Purifier
Никогда не использовать unserialize() с данными от пользователя
Валидировать все пользовательские данные на входе
Использовать CSP (Content Security Policy) для ограничения выполнения скриптов
6. A06:2025 — Insecure Design (Небезопасное проектирование)
Статус: Спустился на две позиции
Уязвимости, заложенные на этапе проектирования архитектуры приложения.
Пример для Konoden:
Отсутствие разделения административной и публичной частей
Хранение конфиденциальных данных в незащищённых JSON-файлах
Отсутствие логирования действий администратора
Что делать:
Проектировать с учётом безопасности с самого начала
Разделять публичную и административную зоны
Логировать все важные действия (вход в админку, изменение данных, удаление заявок)
7. A07:2025 — Authentication Failures (Сбои аутентификации)
Статус: В топ-10
Проблемы с аутентификацией пользователей и управлением сессиями.
Пример для Konoden:
Слабая политика паролей (admin/admin)
Отсутствие защиты от брутфорса
Незащищённые сессионные куки (без HttpOnly, Secure)
Что делать:
Требовать сложные пароли (минимум 8 символов, цифры, буквы в разных регистрах)
Реализовать блокировку после 5 неудачных попыток входа (на 15 минут)
Использовать HttpOnly и Secure флаги для сессионных кук
Внедрить двухфакторную аутентификацию для админ-панели
8. A08:2025 — Software or Data Integrity Failures (Сбои целостности ПО и данных)
Статус: В топ-10
Уязвимости, связанные с нарушением целостности программного обеспечения и данных.
Пример для Konoden:
Возможность подменить JSON-файлы через недостаточно защищённые эндпоинты
Отсутствие проверки целостности загружаемых файлов
Что делать:
Проверять целостность загружаемых файлов (хэши, подписи)
Ограничивать типы загружаемых файлов
Хранить данные в защищённом месте с ограниченным доступом
9. A09:2025 — Logging and Alerting Failures (Сбои логирования и оповещения)
Статус: В топ-10
Отсутствие или неэффективность логирования и систем оповещения об инцидентах.
Пример для Konoden:
Не ведётся лог попыток входа в админ-панель
Нет оповещений о подозрительной активности
Нет системы мониторинга доступности сайта
Что делать:
Логировать все попытки входа (успешные и неудачные)
Настроить оповещения (email/Telegram) о подозрительной активности
Использовать систему мониторинга (например, UptimeRobot)
10. A10:2025 — Mishandling of Exceptional Conditions (Неправильная обработка исключительных условий)
Статус: 🆕 Новая категория в 2025
Некорректная обработка ошибок и исключительных ситуаций.
Пример для Konoden:
Вывод подробных ошибок PHP на экран (раскрывает структуру приложения)
Отсутствие обработки ошибок при работе с JSON-файлами
Падение приложения при неожиданных данных
Что делать:
Отключить display_errors на продакшене
Использовать error_log() для записи ошибок в лог-файл
Обрабатывать все исключения через try-catch
Показывать пользователю только дружественные сообщения об ошибках
ЧАСТЬ 2: СПЕЦИФИЧЕСКИЕ УЯЗВИМОСТИ ДЛЯ ТЕХНОЛОГИЙ ПРОЕКТА
Уязвимости PHP (2025–2026)
PHP остаётся одним из самых популярных языков для веб-разработки, но его популярность также делает его частой целью для кибератак.
CVE	Описание	Версии	Статус
CVE-2026-6735	Неправильная санитация пользовательских данных позволяет выполнить XSS через PHP-FPM статус-страницу	PHP 8.2 < 8.2.31, 8.3 < 8.3.31, 8.4 < 8.4.21, 8.5 < 8.5.6	Актуально
CVE-2026-6722	Use-after-free в SOAP-обработке может привести к RCE	PHP 7.4.33-1+deb11u11	Фиксится обновлением
CVE-2025-14179	Ошибка в обработке имён кодировок в mbstring	Разные версии	Фиксится обновлением
CVE-2025-1735 / CVE-2025-6491	Потенциальная SQL-инъекция при неправильной обработке NULL-значений	Разные версии	Фиксится обновлением
CVE-2025-1220	Проблемы с parse_url() — может привести к обходу проверок доступа	Разные версии	Фиксится обновлением
⚠️ Критическое предупреждение:
Никогда не используйте unserialize() с данными, полученными от пользователя. Это может привести к PHP Object Injection и Remote Code Execution.
В проекте Konoden мы НЕ используем unserialize() — все данные хранятся в JSON и парсятся через json_decode().

ЧАСТЬ 6: РЕКОМЕНДАЦИИ ПО БЕЗОПАСНОЙ РАЗРАБОТКЕ
1. Принцип наименьших привилегий
Каждый пользователь, процесс и система должны иметь только те привилегии, которые необходимы для выполнения их функций.
Для Konoden:
Администратор имеет доступ только к админ-панели
Пользователи (посетители) не имеют доступа к данным других пользователей
PHP-процессы имеют доступ только к необходимым папкам
2. Защита по умолчанию
Система должна быть безопасной "из коробки", без необходимости дополнительной настройки.
Для Konoden:
По умолчанию все данные защищены
По умолчанию все формы требуют CSRF-токен
По умолчанию все выводы экранируются
3. Глубокая защита (Defense in Depth)
Использование нескольких слоёв защиты, чтобы если один слой будет пробит, другие остались.
Для Konoden:
Валидация данных на клиенте + на сервере
Экранирование вывода + CSP
HTTPS + HSTS
Пароль + двухфакторная аутентификация
4. Безопасность по умолчанию (Fail Secure)
В случае ошибки система должна переходить в безопасное состояние, а не в уязвимое.
Для Konoden:
При ошибке валидации — отказ в доступе, а не пропуск
При ошибке базы данных — показ общей ошибки, а не деталей
При истечении сессии — выход из системы
5. Разделение привилегий
Разные функции должны выполняться разными пользователями/процессами.
Для Konoden:
Публичная часть и админ-панель — разные зоны
Чтение и запись данных — разные операции с разными проверками
Гость и авторизованный пользователь — разные права
